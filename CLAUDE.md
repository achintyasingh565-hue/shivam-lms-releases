# CLAUDE.md — System instructions for Shivam Enterprises LMS

These are the standing rules for working in this repository. Follow them in every
interaction with this project.

## Project shape

Electron desktop app (macOS + Windows) for loan management, with Supabase cloud
sync and GitHub-based auto-updates. The frontend is authored as small source
modules under `src/` and assembled by a build step into single artifacts the
Electron shell loads.

## 1. Architecture & editing rules

- **NEVER edit `index.html` or `app.bundle.js` directly.** They are auto-generated
  build artifacts, regenerated by `npm run build`. Any manual edit will be
  overwritten and is a mistake.
- **All frontend edits go inside `src/`:**
  - `src/app/` — application JavaScript modules
  - `src/views/` — HTML view partials (inlined into `index.html` at build time)
  - `src/styles/` — CSS
- **New JS modules in `src/app/` MUST be sequentially numbered** (e.g.
  `26-[feature-name].js`). The build concatenates modules in filename order, so the
  number fixes load order. Do not skip, reuse, or omit the number.
- **The app runs in one SHARED GLOBAL SCOPE.** All `src/app/*.js` files are
  concatenated into a single classic script, so functions and variables are plain
  globals called by name (including from inline HTML `on*` handlers). Do **NOT**
  use ES6 `import`/`export` for internal application logic.
- **CSP blocks inline `<script>` blocks.** Any script must be an external `'self'`
  file (see `src/theme-boot.js`); never inline a `<script>` block in the HTML.
- **Financial math: flat MONTHLY interest only, with `Math.round()` on every INR
  amount** to prevent floating-point drift:
  - `tint = Math.round(p * r / 100 * n)`  (total interest)
  - `tpay = Math.round(p) + tint`          (total payable)
  - `emi  = Math.round(tpay / n)`          (monthly instalment)
  - `recomputeLoan()` reads the stored `tpay`/`emi`; it must not silently re-derive
    the contract terms.

## 2. Build & test commands

- **After modifying any file in `src/`, always run `npm run build`** to re-assemble
  `index.html` + `app.bundle.js`.
- **Run `npm test` before staging changes.** It executes `tests/audit.js`,
  `tests/e2e-verify-fixes.js`, and `tests/cloud-logic.js`.
- For a fuller check: `node tests/health-check.js` must report **10/10 with zero JS
  errors**, and the end-to-end tests (`e2e-restructure.js`, `e2e-snapshot-restore.js`,
  `e2e-sync-indicator.js`, `e2e-offline-persist.js`) should pass.

## 3. Autonomous Git workflow

When asked to build a feature or fix a bug:

1. Make the code changes inside `src/`.
2. Run `npm run build`.
3. Run `npm test`.
4. **If tests PASS:** stage the changes, write a concise commit message, and
   `git push` to trigger the GitHub Actions CI/CD pipeline (which builds and
   publishes the macOS + Windows apps). Do not wait for manual approval to push
   when tests pass.
5. **If tests FAIL:** do not push. Fix the failure, or report it clearly.
6. **Bump `version` in `package.json`** for any change intended to ship — the
   auto-update pop-up only fires when the published version is higher than the
   installed one.

## Do NOT touch without an explicit instruction

- `src/app/24-firm-details.js`
- The conflict-detection logic in `src/app/20-cloud-sync.js`
  (`cloudConflictDecision`, `cloudGuardSave`)
- Recovery-password / OS-keychain / session code (`secureEncrypt`/`secureDecrypt`,
  `saveSession`/`loadSession`, `ensureCryptoKey`)

## Secrets — never commit or paste into chat

- **Safe to ship:** the Supabase *publishable* key and project URL (the database is
  locked by Row-Level Security — verified: anonymous read returns `[]`, anonymous
  write is rejected with Postgres `42501`).
- **Never commit or share:** the Supabase *secret*/service-role key, the database
  password, the WhatsApp access token, and the GitHub Personal Access Token.
